# Гейт платных модулей — включение

**Состояние на 16.07.2026: код задеплоен и проверен, гейт СПИТ.**
Модули сейчас отдаются всем — ровно как до правки. Чтобы закрыть дыру, нужно
одно действие на сервере. Деплой и git при этом не нужны.

## Что было сломано

Сервер отдавал HTML модуля кому угодно, а `role-guard.js` лишь накрывал уже
присланный текст оверлеем. Проверено на проде: `curl` без всякой авторизации

через `curl`, `Ctrl+U`, отключение JS или `document.getElementById('rg-wall').remove()`.

## Что сделано

Файлы модулей переехали в `protected/modules/` — туда закрыт HTTP-доступ
(проверено: 403 на все пути). Отдаёт их `api/module.php` через `readfile()`,
то есть чтением с диска, и только после проверки подписанной куки.
Адреса модулей снаружи не изменились.

## ВКЛЮЧИТЬ (одна команда)

Зайти по SSH на Hostinger и создать ключ подписи **над** `public_html` — там же,
где лежит `groq.key`:

```bash
head -c 48 /dev/urandom | base64 > ~/domains/dispatch4you.com/session.key
chmod 600 ~/domains/dispatch4you.com/session.key
```

Всё. Гейт включился немедленно, без деплоя.

Нет SSH — то же самое через файловый менеджер hPanel: создать файл
`session.key` в `domains/dispatch4you.com/` (рядом с `public_html`, НЕ внутри),
вписать одну длинную случайную строку (минимум 32 символа — короче код
намеренно не примет).

## ОТКАТИТЬ (мгновенно)

```bash
mv ~/domains/dispatch4you.com/session.key ~/domains/dispatch4you.com/session.key.off
```

Гейт снова уснёт, модули начнут отдаваться всем — как сегодня. Без деплоя,
без git, без ожидания. Это и есть рубильник.

## Проверить после включения

Обязательно **под реальным студенческим аккаунтом** — это единственный путь,
который я проверить не смог (нужен вход через Google, а он только руками).

1. **Чужой не проходит.** В приватном окне (или `curl`) открыть
   `https://dispatch4you.com/pages/doc-module-7-complete.html`
   → должен появиться экран входа, а в исходнике страницы (`Ctrl+U`) **не должно
   быть текста курса**. Это главная проверка — ради неё всё делалось.

2. **Свой проходит.** Войти под аккаунтом студента (роль `student` в Firestore),
   открыть тот же модуль → должен открыться как обычно, максимум с коротким
   спиннером на первом заходе (это загрузчик меняет токен на куку).

3. **Не-студент видит пейволл.** Войти под зарегистрированным без оплаты →
   модуль 1 открыт, модули 2–12 показывают «Контент для студентов».

**Если что-то не так — сразу выполнить откат выше**, и напишите мне: разберусь
по логам. Ошибка в гейте выкидывает платящих клиентов, поэтому откат — первое
действие, а разбор — второе.

## Как это работает

```
Запрос /pages/doc-module-7-complete.html
        ↓ .htaccess перенаправляет
   api/module.php
        ↓
   Нет session.key?  → отдать модуль (гейт спит = сегодняшнее поведение)
        ↓ есть
   Кука валидна и роль подходит?  → отдать модуль
        ↓ нет куки                    ↓ роль не та
   страница-загрузчик            пейволл (403)
        ↓ берёт токен из Firebase SDK
   POST /api/session.php
        ↓ проверяет токен у Google (identitytoolkit)
        ↓ читает роль из Firestore токеном САМОГО пользователя
   ставит HttpOnly-куку → перезагрузка → модуль отдан
```

Куку подписываем HMAC-SHA256 ключом из `session.key`. Подделать нельзя, не зная
ключа. Кука `HttpOnly` — её не украсть через XSS. Живёт 2 часа, фронт продлевает
её при каждом обновлении токена Firebase (раз в час).

Проверка токена идёт REST-запросом к Google, а не локальной проверкой подписи
JWT — потому что Composer в проекте нет, значит нет и библиотеки. Запрос идёт
раз в сессию (~раз в час на пользователя), не на каждую страницу, так что цена
не важна.

Роль читается из Firestore **токеном самого пользователя** — сервисный аккаунт
и его ключ не нужны вовсе.

## Что доделать потом

1. **Перевернуть на fail-closed.** Сейчас нет ключа → гейт спит. Это осознанный
   компромисс на время выкатки: ошибка в конфиге возвращает старое поведение, а
   не выкидывает оплативших. Но означает и обратное: случайно удалили ключ —
   платный контент тихо открылся, и никто не заметит. Когда гейт проверен на
   живом студенте, поменять в `api/module.php` строку
   `if ($secret === null) { d4y_serve($file); }` на отдачу 503. Помечено
   комментарием `ponytail:` прямо в коде.

   Firebase. Вопрос один: может ли любой залогиненный читать/писать коллекцию
   `users`? Если да — роль `student` можно себе выписать самому, и гейт обойдут
   легально. Анонимный листинг я проверил, он закрыт (403), но авторизованный —
   нет.

3. **Админка** (`pages/admin.html:2075`) пускает по строке в localStorage. Гейт её

4. **role-guard.js оставлен как есть** — теперь он чисто косметический слой (UX,
   спиннер). Реальную проверку делает сервер. Удалять его не нужно, но и полагаться
   на него больше нельзя.
