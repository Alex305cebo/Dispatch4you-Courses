## Error Handling

### Graceful Degradation
All Firestore save operations use try-catch blocks:
- **On error:** Log to console, continue execution
- **User experience:** Not interrupted by save failures
- **Debugging:** Clear error messages in console

### Validation
SessionManager validates all messages:
- Type must be 'user', 'broker', or 'system'
- Content must be non-empty string
- Timestamp automatically added if missing

## Data Structure

### Message Format in Firestore
```javascript
{
  type: 'user' | 'broker' | 'system',
  content: string,
  timestamp: Firestore.Timestamp
}
```

### Session Document Structure
```javascript
brokerSessions/{sessionId}/
  - uid: string
  - scenario: string
  - brokerName: string
  - startedAt: Timestamp
  - completedAt: Timestamp | null
  - messages: array of message objects  // ← Task 8.2 populates this
  - metrics: object | null
  - xpAwarded: number
  - duration: number (seconds)
```

## Performance Considerations

### Real-time Saving
- Messages saved asynchronously (await)
- Does not block UI updates
- Chat continues even if save fails

### Firestore Operations
- Uses `arrayUnion` for atomic array updates
- No race conditions on concurrent saves
- Efficient batch operations for multiple messages
